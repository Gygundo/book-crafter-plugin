**Section 1: On Invocation**
Receive via $ARGUMENTS:
- Project directory path
- Edit mode: "full" (all three passes) or "revision" (targeted re-edit of specific chapters)
- Chapters to edit (optional, for revision mode -- list of chapter numbers)

Steps:
1. Read `[project_directory]/book-dna.md` for voice profile summary, theological framework, chapter map, running themes, key terms
2. Read `[project_directory]/voice-profile.md` for detailed voice rules (Tone, Sentence Patterns, Vocabulary Use/Avoid, Emphasis Techniques, Anti-Patterns, Theological Framework)
3. Read `[project_directory]/chapter-outline.md` for chapter count, momentum positions, and cross-chapter connections
4. Count chapters: `ls [project_directory]/drafts/ch*-draft.md | wc -l`
5. Determine editing strategy:
   - 15 or fewer chapters: Editor reads all chapters directly (no subagents for Pass 1)
   - 16+ chapters: Use chapter-editor subagents with rolling window for Pass 1
6. Create output directories: `mkdir -p [project_directory]/edited [project_directory]/reports`

**Section 2: Pass 1 -- Voice Consistency + Theological Guardrails (EDIT-01, EDIT-03)**
Purpose: Normalise each chapter's voice against the profile. This pass runs FIRST because subsequent passes need voice-normalised text.

For each chapter (parallel via subagents if 16+ chapters, sequential otherwise):
1. **Vocabulary audit**: Scan for words/phrases from the Avoid list. Case-insensitive matching. Flag every occurrence with approximate line location and the specific Avoid rule violated.
2. **Sentence length distribution**: Count words per sentence (split on `.`, `!`, `?` boundaries). Calculate average sentence length and fragment percentage (sentences with 8 or fewer words). Compare against voice profile targets (12-18 average, frequent 3-8 word fragments). Flag chapters whose average deviates by more than 4 words from target.
3. **Anti-pattern detection**: Check for each anti-pattern listed in the voice profile's Anti-Patterns section AND the Theological Framework section (for theological books). Specific checks:
   - Academic hedging patterns: "some scholars believe", "it could possibly mean", "there is a view that", "it is important to note"
   - Religious cliches: "God won't give you more than you can handle", "everything happens for a reason"
   - Passive voice: Flag sentences where active voice is possible
   - Performance-based guilt: "you need to pray more", "you should be doing"
   - Cessationist framing: implying gifts have ceased
   - AI voice indicators: overly balanced, hedged, or neutral tone; "various perspectives"; "many Christians believe"
4. **Tone normalisation**: For each flagged passage, rewrite to match the voice profile while preserving the argument and scripture references. Replace Avoid-list vocabulary with Use-list alternatives. Convert passive to active voice. Sharpen hedged language into direct declarations.
5. **Theological guardrail check (EDIT-03)**: If voice profile contains a Theological Framework section, validate each chapter against it. Flag content that contradicts the framework (e.g., performance-based language contradicting grace-over-law, cessationist framing contradicting supernatural-is-active-today). For non-theological books, skip this sub-check.

Output per chapter: Save to `[project_directory]/edited/ch[NN]-pass1.md` with the voice audit metadata block at the end:
```markdown
<!-- VOICE AUDIT
chapter: [N]
vocabulary_violations: [count]
  - Line ~[N]: "[phrase]" ([Avoid rule])
avg_sentence_length: [number]
fragment_percentage: [number]%
anti_patterns_found:
  - Line ~[N]: [description] ([pattern name])
theological_flags: [list or "none"]
changes_made: [count]
severity: clean | minor | significant
-->
```

**Section 3: Pass 2 -- Flow and Transitions (EDIT-02)**
Purpose: Ensure every chapter ending connects naturally to the next chapter opening. This pass is ALWAYS sequential because each transition depends on both chapters.

Critical rule: This pass ONLY modifies the final 2-3 paragraphs and first 2-3 paragraphs of chapters. Never change the body.

For each consecutive chapter pair (Ch1->Ch2, Ch2->Ch3, ...):
1. Read the final 2-3 paragraphs of Chapter N from `edited/ch[NN]-pass1.md`
2. Read the first 2-3 paragraphs of Chapter N+1 from `edited/ch[N+1]-pass1.md`
3. Read both chapters' momentum positions from the outline
4. Check: Does the ending plant a seed that the opening picks up?
5. Check: Does the momentum position shift feel right? (Foundation->Building = escalation, not topic change)
6. If the transition is jarring, rewrite the chapter ending and/or opening to create a bridge. Preserve core argument. Only modify transitional language.
7. IMPORTANT: Re-read the voice profile before rewriting any text. Transition rewrites must match the voice, not just the logical connection.

Output: Save each chapter to `[project_directory]/edited/ch[NN]-pass2.md`. If no changes were needed for a chapter, copy the pass1 file as-is to pass2.

Write `[project_directory]/reports/flow-report.md`:
```markdown
# Flow Report: [Book Title]

| Transition | Status | Action Taken |
|-----------|--------|--------------|
| Ch 1 -> Ch 2 | smooth | none |
| Ch 2 -> Ch 3 | jarring | rewrote Ch 2 ending to bridge to Ch 3's argument |
```

**Section 4: Pass 3 -- Cross-Chapter Validation (EDIT-04)**
Purpose: Build indexes across all chapters and validate consistency. This pass reads all chapters but works primarily from extracted indexes, not full text (important for context limits).

Steps:
1. **Term index**: Extract key terms and jargon from all chapters. Flag terms used inconsistently (different definitions, spellings, capitalisation drift). Cross-reference against Book DNA Key Terms section.
2. **Reference validation**: Find all forward references ("we'll explore this in chapter 7", "as we saw in chapter 3", "later in the book", "as we discussed earlier") using pattern matching. Verify each referenced chapter delivers on the promise. Flag vague references ("as we saw earlier") and recommend specifying the chapter.
3. **Scripture consistency**: Verify the same scripture passage is not quoted with different wording in different chapters. All quotes should use the same translation (default NKJV per voice profile Scripture Handling section).
4. **Theme tracking**: Cross-reference against Book DNA Running Themes section. Verify themes are introduced, developed, and climaxed in the chapters specified in the chapter map.

Output: Save final edited chapters to `[project_directory]/edited/ch[NN]-final.md` (copy from pass2 if no changes needed in this pass).

Write `[project_directory]/reports/consistency-report.md` using this exact structure:

**Section 5: Rolling Window Pattern (EDIT-05)**
For books with 16+ chapters, use chapter-editor subagents for Pass 1:

Window composition per subagent:
- Current chapter (full text, the focus of editing)
- Previous chapter's final 500 words (for context)
- Next chapter's first 500 words (for context)
- Full voice profile and Book DNA (always included)

Subagent invocation: Use the `chapter-editor` subagent from `${CLAUDE_PLUGIN_ROOT}/agents/chapter-editor.md`. Each receives:
- Project directory path
- Chapter number to edit
- Edit pass: "voice" (Pass 1 only for subagents)
- Voice profile path
- Book DNA path

Pass 1 subagents can run in parallel (each chapter is independent for voice checking).
Pass 2 is ALWAYS handled by the main editor skill sequentially (transitions need chapter pairs).
Pass 3 is ALWAYS handled by the main editor skill (needs cross-chapter indexes).

**Section 6: Revision Mode**
When invoked with mode "revision" and a list of chapter numbers:
1. For each chapter to revise, read the new draft from `drafts/ch[NN]-draft.md`
2. Run Pass 1 (voice consistency) on the revised chapter only
3. Run Pass 2 (flow/transitions) on the revised chapter AND its immediate neighbours (ch[N-1] and ch[N+1]). Limit to ONE hop -- do not recursively check beyond immediate neighbours.
4. Run Pass 3 (cross-chapter validation) on affected references only -- scan the revised chapter for forward/backward references and validate those specific references
5. Update the consistency report with the revision results

**Section 7: Output Summary**
After all passes complete, return a summary to the orchestrator:
```
Editing complete for [Book Title].
Chapters edited: [N]
Voice consistency: [Clean/Minor/Significant] ([X] issues found, [Y] auto-resolved)
Transitions: [X]/[N-1] smooth
Cross-references: [X] validated, [Y] flagged
Report: [project_directory]/reports/consistency-report.md
```

**Section 8: Anti-Patterns**
- Do NOT edit body text during the flow pass (Pass 2) -- only touch final/first paragraphs
- Do NOT run passes in parallel -- passes MUST be sequential (Pass 2 needs Pass 1 output, Pass 3 needs Pass 2 output)
- Do NOT overwrite original drafts -- always write to `edited/` directory
- Do NOT produce subjective voice flags -- every flag must cite a specific rule from the voice profile (vocabulary violation, anti-pattern match, sentence length deviation)
- Do NOT modify Book DNA, voice-profile.md, chapter-outline.md, or any shared file
- Do NOT spawn subagents from within subagents -- if running as a chapter-editor subagent, work directly
- Do NOT recursively check beyond one hop during revision adjacency checks
